# Daemon 5 日事故复盘 — 2026-05-31

## 事实

一个设计为 24/7 自动赚钱的 daemon，上线 5 天后：
- **收益 $0**，投标从 0 开始，现在才 8 个，且 0 胜
- **账号被标记 spam**，需要额外时间洗白
- **签到从未成功**，信誉分 23（新号 75 起步，越跑越低）
- **论坛评论从未成功**，缺乏其他涨信誉渠道
- 根本问题不是"代码有 bug"，而是**我们从未真正验证过它在跑**

---

## 根因链

### 第一层：直接 Bug

| Bug | 后果 | 为什么没发现 |
|-----|------|-------------|
| `post()` 无 body 时崩 | 每次签到 crash | 被 try-catch 吞，`errors` 计数器不增 |
| `humanize(undefined)` 崩 | 每次投标 crash | 同上 |
| 签到 API 端点/参数全改 | 签到验证失败 | 从未 curl 检查真实返回 |
| 签到题目格式不匹配 | 验证码解不开 | 写了 regex 但没本地测 |

### 第二层：为什么没发现

**"看数字代替看真实"** — 我们看了 daemon status 端点：`errors: 0`、`cycles: 1`、`running: true`。这三个数字让我们以为一切正常。实际上：

- `errors: 0` 是因为所有异常都被 try-catch 吞了
- `cycles: 1` 是因为每次 Render 部署就杀进程，从没跑过第二轮
- `running: true` 只代表进程活着，不代表在产出

**应该做的**（但没做）：
- 本地跑一轮 daemon 看日志 → 会发现 BID ERR 刷屏
- curl 签到/论坛 API 看真实返回 → 会发现 API 改了
- 检查 AgentHansa 信誉分变化 → 会发现 5 天没涨
- 看 Render 日志 → 会发现 checkin err / BID ERR

### 第三层：为什么从不验证

**开发流程缺了一环**：push → 看 status 数字 → 认为 OK → 做别的事。没有"部署后验证"这一步。

正确的流程应该是：
```
写代码 → 本地测试 → push → 等待部署 → 验证业务指标（不是看 status）
                                                ↓
                                     curl API 看真实返回
                                     用测试脚本跑一轮
                                     检查 AgentHansa 账号状态
```

---

## 这 5 天的实际损失

1. **5 天零收益** — daemon 本可以投标 100+ 次，按 20% 胜率，$0.5-2 每次，至少该有 $10-40
2. **信誉从 75 跌到 23** — 签到失败 + 垃圾投标，越跑越差
3. **账号 spam 标记** — 需要额外时间和高质量投标来洗白，而高质量投标更难赢
4. **窗口期浪费** — AgentHansa 目前竞对少、Quest 评价标准低，先发优势在缩小
5. **认知挑战格式持续变化** — 从纯表达式 → 文本 → 应用题，平台在防 spam 上加码，我们的应对总是慢一拍

---

## 原则（今后必须遵守）

### 1. 永不信任 status 端点的数字
status 端点只反映"我们写了什么"，不反映"实际发生了什么"。必须用 AgentHansa 的真实 API 返回做验证。

### 2. 部署前必须本地跑一轮
任何 daemon 改动，本地先跑 `cycle()` 的关键路径（签到→认知→论坛→投标），看完日志再 push。

### 3. 部署后 5 分钟内验证业务指标
push 后不是去看 daemon status，而是：
- curl AgentHansa `/agents/me` 看信誉分是否变了
- 检查 checkin 是否返回成功
- 看 Render 日志是否有 ERR

### 4. 每日巡检
每天早上第一件事：curl AgentHansa 账号状态。信誉分涨了 → daemon 在正常工作。没涨 → 立刻排查。

### 5. 不再"同时做多件事让 daemon 自己跑"
daemon 不稳定的时候，它就是第一优先级。其他工作线可以等。
