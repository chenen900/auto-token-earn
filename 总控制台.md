9. **禁止覆盖已有文件** — 任何 Write/Edit 操作前，必须先 Read 目标文件（及目标目录下同名文件），确认是否已存在。若已存在，判断是"同一件事的扩展"还是"完全不同的事"。若扩展，在原内容基础上追加；若不同，加前缀抬头区分。绝不覆盖已有的、有意义的内容，避免造成损失
10. **参考库文件加前缀** — 在第三方参考库目录（reference/）中新增文件时，文件名必须加前缀（如 `CLAUDE-BRIDGE.md`、`SKILL-REVERSE.md`），避免与库自带的 CLAUDE.md / README.md 冲突
11. **永不信任自己的推测，必须验证真实数据** — 来自 [POSTMORTEM](auto-token-earn/POSTMORTEM.md) 5天零收益事故教训。status端点只反映"我们写了什么"，不反映"实际发生了什么"。所有判断必须基于外部真实数据（API返回、实际日志、真实文件内容），不得相信自己的记忆、推测、或内部状态数字。不要想当然，不要自以为知道
12. **部署后必须验证业务指标** — 同上来自 POSTMORTEM。任何自动化系统上线后，不得仅检查status端点（`errors:0`、`running:true`），必须用第三方视角验证：curl真实API看返回、检查账号信誉/收益是否真的在涨、看日志是否有异常。push后5分钟内完成验证，未通过则立刻排查
13. **找到一个根因 ≠ 找到所有根因** — 来自 [GOVERNANCE](GOVERNANCE.md)。当系统长期静默失败时，每个与外部交互的模块都是独立故障点。修好第一个bug后必须追问：同一时期内还有哪些模块也依赖外部API？那些API是否也变了？
14. **每次外部API更新都是潜在断裂点** — 第三方API随时可能改格式。代码不是"写对一次就永远对"。每次排查故障时，假设API可能已经变了，用curl重新确认返回格式
15. **输出双格式** — 给人看的用 Word（.docx），给 AI 看的用 Markdown（.md）。人读 Word 舒服，AI 读 Markdown 高效。两者同步更新。

### 📋 操作检查清单（来自 GOVERNANCE + POSTMORTEM + DEBUGGING_LOG）

**代码修改后本地验证：**
- daemon 改动 → 本地跑一轮 cycle（签到→认知→论坛→投标），看完整日志再 push
- API/服务改动 → 本地跑全部测试用例（至少 5 个端点），确认语法不报错
- 任何代码 → `node -e "require('./xxx')"` 做基础语法检查

**错误处理铁律（来自 DEBUGGING_LOG）：**
- try-catch 内必须打印完整堆栈 `e.stack`，不能只 print `e.message`
- 所有工具函数必须防御 null/undefined 输入（`if (!input) return default`）
- 关键路径的每个 catch 都应该让人能定位到具体行号

**部署后 5 分钟验证（不可跳过）：**
1. 等 Render 部署（30-60s）
2. `curl /health` 确认 version 更新
3. `curl /daemon/status` 确认 uptime < 120s（新进程）
4. 等一轮周期（约 90s）→ 确认 lastCheckin/lastCognitive/submissionsToday 有变化
5. `curl AgentHansa /api/agents/me` 确认信誉分是否真的在涨

**禁止的操作：**
- ✗ 不本地测试就 push daemon/API 改动
- ✗ push 后不检查线上状态就去做别的事
- ✗ 同时 push 多个工作线的改动（混在一起无法定位问题）
- ✗ 相信 `errors: 0` 就是没问题

**事故响应（来自 GOVERNANCE 第六章）：**
当 `errors > 0` 或指标连续 12 小时不增长 →
1. 暂停所有其他工作线 → 2. 本地复现 → 3. curl 所有相关 API 比对格式 → 4. 修复+本地验证+push+线上验证 → 5. 确认恢复后更新 POSTMORTEM.md

**完整性自检（来自用户反馈 — 多次"搭好框架但细节不全"）：**
- [ ] 这个功能覆盖了所有可能的情况吗？（不只是 happy path）

- [ ] 和已有系统接上了吗？（前端↔后端↔CI↔daemon）
- [ ] 权限检查加了吗？CI env var 写了吗？
- [ ] 用户能不能直接用？

**发布前铁律（来自 feedback-resilience）：**

- [ ] 依赖的文件/变量有 try/catch 保护
